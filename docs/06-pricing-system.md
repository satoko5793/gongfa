# 06 残卷定价系统

## 目标

这套系统解决 4 个问题：

- 价格不能低于分解底线
- 好卡长期要比差卡贵
- 图鉴价值和佩戴价值要同时成立
- 市场价允许明显波动，但不能把整套价格体系打乱

最终单卡价格公式：

`最终价 = max(分解底价, 图鉴价, 佩戴价) * 市场系数`

如果管理员设置了手动价，则：

`最终价 = 手动价`

但自动价解释仍保留在 `pricing_meta`，便于后台查看。

## 1. 分解底价

按分解残卷的 2 倍作为硬底价：

- 绿卡：`50 x 2 = 100`
- 蓝卡：`60 x 2 = 120`
- 紫卡：`60 x 2 = 120`
- 橙卡：`80 x 2 = 160`
- 红卡：`100 x 2 = 200`
- 金卡：`200 x 2 = 400`

说明：

- 自动价永远不能低于这个底线
- 市场系数再低也不能击穿底价

## 2. 官方参考配置

配置文件：`backend/src/config/catalog-config.js`

当前定价不再把“当前导入数据里观察到的最大值”作为主标准，而是优先使用配置表：

- `LEGACY_CAPS`
  - `attack_max`
  - `hp_max`
  - `fire_total_max`
  - `calm_total_max`

用途：

- 图鉴价按理论满攻击、满血量归一化
- 佩戴价按理论满走火、满气定归一化
- 同一 `legacy_id` 的价格判断会更稳定，不会因为导入批次不同而飘

兼容策略：

- 如果某个 `legacy_id` 还没写进 `LEGACY_CAPS`
- 系统会退回到 `observed_fallback`
- 也就是只对未配置的类型临时使用导入数据中的最大值

`pricing_meta` 中会记录：

- `reference_caps`
- `reference_source`

## 3. 图鉴价

图鉴价只看攻击和血量，不看词条。

### 图鉴评分

- `attack_rate = 当前攻击 / theory_attack_max`
- `hp_rate = 当前血量 / theory_hp_max`
- `atlas_score = attack_rate * 0.72 + hp_rate * 0.28`

附加奖励：

- 满攻击额外加分
- 高血量额外加分
- 金卡对满攻、高血、双满有更强锚点保护

### 图鉴锚点

- 蓝卡满属性：`1000`
- 紫卡非双满最高约：`600`
- 橙卡非双满最高约：`800`
- 红卡非双满最高约：`3000`
- 金卡满属性：`50000`

红橙双满例外：

- 紫卡双满可抬到 `1500`
- 橙卡双满可抬到 `2500`
- 红卡双满可抬到 `12000`

金卡额外锚点：

- 满攻击金卡：`8000`
- 满攻击高血金卡：`12000`
- 双满金卡：`50000`

## 4. 佩戴价

佩戴价主要看词条，同时吃一部分属性修正。

### 词条优先级

- 走火高于气定
- 双词条高于单词条
- 同类型词条里，高数值高于低数值

### 词条分类

- `none`
- `single_fire`
- `single_calm`
- `dual_mixed`

说明：

- 当前业务里双词条只存在一种合法组合：`走火 + 气定`
- 不存在“双走火”或“双气定”定价分支

### 佩戴价锚点

- 金卡单走火：基础 `6000`
- 金卡单气定：基础 `6000`
- 金卡双词条：基础 `12000`

红橙紫卡说明：

- 紫卡图鉴价同样改成“只有精确双满才吃高锚点”
- 橙卡词条当前基本不单独抬价，实际售价几乎只看攻击和血量
- 红卡单词条当前基本不单独抬价，实际售价基本只看攻击和血量
- 红卡只给双词条保留少量溢价，用来体现“有一点影响”，但不会喧宾夺主

说明：

- 走火整体优先级高于气定
- 同词条类型下，属性越高越贵
- 金卡当前按官方参考表以 `3.0` 作为满词条
- 金卡里 `2.5+` 已视为稀缺高词条，会额外加价
- `2.7`、`2.8` 属于明显更少见的高档位，价格会再抬一层
- 同样是高词条，走火加价高于气定

## 5. 市场系数

市场系数用于动态涨跌价，允许明显波动。

当前区间：

- 最低：`0.5`
- 最高：`4.0`

热度加成参考：

- 最近 7 天确认订单数
- 最近 30 天有效订单权重
- 最近 30 天独立买家数

冷却减成参考：

- 长时间无人购买
- 有成交但热度持续走低

说明：

- `confirmed` 权重最高
- `pending` 低权重计入
- `cancelled` 不计入热度

## 6. 单调校正

同一 `legacy_id` 内，自动价还会做一次顺序校正：

- 先把商品拆成三组：
  - 双词条
  - 单词条
  - 普通卡
- 顺序上始终是：双词条 > 单词条 > 普通卡
- 但压价链只在各自组内生效，不跨组互相挤压
- 当前最小价差已放宽到 `50`
- 只做顺序修正，不强行拉出离谱价差

这样可以防止：

- 热门弱卡因为短期订单过多，价格冲到强卡上面
- 冷门强卡因为短期无成交，被压到弱卡下面
- 低词条卡被好普通卡或普通卡链路异常压穿
- 双词条被单词条链路连续拖价

## 7. 手动价

管理员可以人工覆盖单卡价格：

- 写入 `manual_price_quota`
- 最终售卖价使用手动价
- 自动定价解释仍保留在 `pricing_meta`
- 随时可以“恢复自动价”

适用场景：

- 临时清仓
- 活动价
- 定向运营测试

## 8. 套餐 SKU

以下商品不再混入单卡自动定价，而是独立做成套餐 SKU：

- 全图鉴
- 满攻击全套图鉴
- 其他后续需要固定价运营的图鉴类套餐

配置来源：

- `BUNDLE_SKU_SEEDS`

落库表：

- `bundle_skus`

规则：

- 套餐 SKU 使用固定价
- 套餐库存可为有限或不限量
- 套餐可独立上下架、改价、改展示顺序
- 套餐下单走正常订单、额度、审计和退款链路

## 9. 当前实现范围

当前版本已经实现：

- 官方参考上限表 `LEGACY_CAPS`
- 套餐种子配置 `BUNDLE_SKU_SEEDS`
- 导入后单卡自动定价
- 下单后自动重算单卡价格
- 取消订单后自动重算单卡价格
- 后台查看底价、自动价、市场系数、主导原因
- 手动改价
- 恢复自动价
- 后台手动触发“重算定价”
- 套餐 SKU 独立展示、独立管理、独立下单

当前仍建议后续补强：

- 把所有正式 `legacy_id` 的理论满值补齐
- 针对不同买家画像做更细的热度分组
- 增加价格走势历史快照
- 增加套餐组合推荐和套餐库存策略
