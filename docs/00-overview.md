# 00 概览

目标：建立一个“功法卡片商城”MVP，先跑通管理员导入卡片、用户扫码绑定、商品展示、额度下单、后台管理的基础闭环。

## 产品定位

这是一个偏运营型的轻商城，而不是记录/对账系统。

业务链路：
1. 管理员通过 `xyzw_web_helper` 或本地文件获取卡片 JSON
2. 在后台导入卡片数据，生成商品
3. 为商品配置展示信息、价格、库存和上架状态
4. 用户扫码登录并绑定咸鱼角色
5. 用户在前台浏览商品，使用站内额度提交订单
6. 管理员在后台查看订单并手工处理发货或确认

## 当前范围

MVP 必做：
- 卡片 JSON 导入
- 商品管理
- 扫码绑定账号
- 商品列表和商品详情
- 额度账户
- 下单与订单管理

MVP 暂不做：
- 真实支付
- 短信登录
- 自动发货
- 复杂营销能力
- 多仓/多库存批次
- 完整 BI 报表

## 角色定义

- 管理员：导入卡片、编辑商品、维护用户额度、处理订单
- 用户：绑定角色、浏览商品、查看额度、提交订单、查看订单状态

## 代码现状（2026-03-14）

现有仓库已经具备两部分可复用资产：

- `frontend/index.html` 中已有卡片 JSON 解析、图片映射、筛选与展示逻辑
- `xyzw_web_helper/` 已具备扫码登录、角色与 token 读取能力，可作为绑定与导入前置工具

现有仓库也有两部分需要替换：

- `docs/` 原先主要描述记录系统，现已改为商城主线
- `backend/` 原先围绕短信登录和参与者记录设计，接下来会切换到商城数据模型

## 目录职责

- `backend/`：商城 API、权限、订单、额度、后台管理
- `frontend/`：商城前台与后台基础页面
- `infra/`：PostgreSQL schema 和部署配置
- `xyzw_web_helper/`：扫码绑定和卡片数据来源，不承担商城核心业务状态

## 当前开发原则

- 先实现最小可运行闭环，不在第一阶段做真实支付
- 商品、额度、订单状态必须由后端持久化，不能继续仅靠浏览器本地存储
- helper 负责“获取和桥接数据”，商城负责“保存和管理业务状态”
- 所有后续开发以本套文档为准
