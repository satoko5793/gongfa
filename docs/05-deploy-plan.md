# 05 开发与部署计划

## 开发阶段

### 阶段 1：文档与模型收敛

目标：
- 明确商城 MVP 边界
- 统一文档、数据库模型、接口设计

交付：
- 更新 `docs/00~05`
- 明确 helper 与商城的职责边界

### 阶段 2：后端基础能力

目标：
- 提供最小可用商城 API

交付：
- 用户绑定登录
- 商品导入与商品管理
- 额度账户和额度流水
- 下单与订单查询
- 管理员订单处理

### 阶段 3：前端基础页面

目标：
- 提供可操作的用户前台和管理员后台

交付：
- 商品列表页
- 绑定页
- 我的页面
- 管理导入页
- 商品管理页
- 用户额度管理页
- 订单管理页

### 阶段 4：联调与验收

目标：
- 跑通导入、绑定、浏览、下单、后台处理的完整链路

验收链路：
1. 管理员导入 JSON
2. 管理员上架商品并设置价格
3. 用户扫码绑定
4. 管理员给用户充值额度
5. 用户下单
6. 管理员确认订单

## 部署目标

- 单台云服务器即可跑通 MVP
- PostgreSQL + Node.js
- Nginx 反向代理
- HTTPS

## 部署结构

- `frontend/` 作为静态资源站点
- `backend/` 提供 API
- `db` 使用 PostgreSQL

建议：
- 前端静态文件由 Nginx 托管
- 后端以单独 Node 进程或容器运行
- 数据库单独容器

## 部署步骤（MVP）

1. 准备云服务器（2 核 4G 足够起步）
2. 安装 Docker 和 Docker Compose
3. 启动 PostgreSQL
4. 配置后端 `.env`
5. 执行 schema 初始化
6. 部署前端静态页面
7. 使用 Nginx 反代 API
8. 配置域名和 HTTPS

## 测试建议

联调时优先手工验证以下场景：
- JSON 导入是否稳定
- 商品上架后前台是否可见
- 扫码绑定是否能拿到正确角色
- 额度不足时下单是否被拒绝
- 取消订单后额度是否回补
