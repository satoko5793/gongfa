# 07 小程序后续计划

## 当前决策

当前阶段先把网页端跑通，不立即切小程序。

原因：

- 现有前后端已经能跑商品、登录、额度、下单、后台管理
- 现在直接转小程序，会把前台界面和交互全部重写，拖慢主线
- 先验证网页端商品展示、登录购买、后台运营是否顺手，更符合当前目标

因此本文件只作为后续路线保留，不纳入当前开发范围。

## 未来切小程序的目标

如果后续确认要长期运营，并且希望用户在微信里更方便打开和购买，可以把“用户端前台”迁移到小程序。

迁移目标：

- 用户端改为微信小程序
- 后台管理继续保留网页端
- 商品、额度、订单、定价规则尽量复用现有后端模型

## 建议的系统形态

推荐采用双端结构：

- 网页后台：
  - 管理员登录
  - JSON 导入
  - 商品管理
  - 额度管理
  - 订单处理
  - 定价调优

- 小程序前台：
  - 商品列表
  - 商品详情
  - 用户登录
  - 额度展示
  - 下单
  - 我的订单

## 现有代码可复用的部分

以下内容未来可以继续复用，不需要推倒重来：

- 数据模型设计
  - `users`
  - `products`
  - `bundle_skus`
  - `user_quota_accounts`
  - `orders`
  - `order_items`
- 定价系统
  - 单卡自动定价
  - 套餐 SKU
  - 手动改价与自动价解释
- 后台运营逻辑
  - 导入
  - 上下架
  - 调额度
  - 处理订单

## 需要重做的部分

以下内容不能直接搬到小程序前台：

- `frontend/` 现有 HTML + JS 页面
- 现有浏览器 `localStorage` 登录态
- 现有基于网页 DOM 的商品详情弹层和筛选交互

也就是说，小程序主要是“重做用户端前台”，不是“重做整个系统”。

## 登录方案建议

如果未来切小程序，建议把用户端登录改成两层：

- 第一层：微信小程序登录
  - 用于识别商城用户
- 第二层：补充游戏身份
  - 可继续使用游戏 ID
  - 或保留 helper / 角色绑定能力

建议不要把当前网页端的“游戏 ID + 密码”直接照搬成小程序唯一登录方案。

更合理的方向是：

- 小程序先用微信身份建立商城账号
- 再补游戏角色信息

## 推荐迁移顺序

### 阶段 1

先把网页端跑通：

- 前台商品页
- 密码注册登录
- 额度购买
- 后台商品管理
- 后台订单管理

### 阶段 2

把后端接口整理成更稳定的用户端 API：

- 登录
- 商品列表
- 商品详情
- 我的额度
- 我的订单
- 下单

### 阶段 3

再单独开一个小程序前台工程：

- 首页商品列表
- 分类与搜索
- 商品详情
- 登录页
- 我的页面
- 订单页

### 阶段 4

根据运营结果决定是否逐步替换网页前台：

- 如果小程序体验更好，就把用户流量导向小程序
- 网页端保留后台和调试入口

## 当前不做的事

为了保证网页主线推进，以下内容现在不做：

- 不新建小程序工程
- 不改现有后端为小程序专用接口
- 不接微信支付
- 不接微信登录
- 不做小程序审核相关适配

## 触发切换条件

满足下面任意两条，再启动小程序计划更合适：

- 网页端已经能稳定跑完整购买流程
- 管理后台已经足够顺手
- 有稳定的真实用户访问需求
- 需要更低的用户使用门槛
- 愿意开始处理微信生态相关配置和审核

## 结论

当前主线不变：

- 继续把网页端跑通
- 把商品、登录、购买、后台做顺

小程序作为下一阶段方案保留，等网页端闭环稳定后再启动。
