# 多炉子仓库与自动发货计划

## 目标

把现在“单账号阵容中心”的能力，继续扩展成：

- 一个商城账号可绑定多个游戏角色
- 多个角色的背包功法可合并成“总仓库”
- 订单完成残卷转入后，系统可识别对应收款记录
- 识别成功后，系统可自动从合适的炉子号发货

这条路线分阶段做，先把“多炉子仓库展示”做稳，再做“自动发货”。

## 当前基础

当前已经有的能力：

- 商城账号已支持绑定游戏角色
- helper 凭证已支持云端保存，能跨设备使用
- 阵容保存 / 一键还原已经在线
- helper 读角色数据、读阵容、执行恢复已经跑通

这意味着：

- 角色绑定链路已经存在
- 云端持久化能力已经存在
- 可以继续往“多绑定”和“自动执行”扩展

## 核心产品方向

### 1. 多炉子仓库

用户在商城里不只绑定一个号，而是可以绑定多个：

- 大号
- 小号
- 仓库号
- 发货号

商城要能直接展示：

- 每个号当前背包有哪些功法
- 某张功法在哪个号上
- 多个号合并后的总库存
- 哪个号更适合发货

用户视角应更像：

- 一个统一仓库
- 多个发货来源
- 不需要理解 helper

### 2. 自动发货

订单创建后，用户先在游戏里转残卷。

系统要能：

- 读取每个绑定炉子号的“接收残卷记录”
- 自动识别这条记录对应哪笔订单
- 识别成功后自动发功法
- 回写订单状态

用户最终感知应是：

- 下单
- 转残卷
- 自动收到功法

## 分阶段实施

## 阶段 1：多炉子绑定与仓库展示

目标：

- 一个商城账号支持绑定多个游戏角色
- 允许用户给每个角色设置用途
- 展示每个角色和合并后的功法库存

### 范围

- 支持多个游戏角色绑定
- 支持角色备注 / 别名
- 支持角色用途：
  - 主号
  - 仓库号
  - 发货号
  - 阵容号
- 支持手动刷新背包功法
- 支持合并库存视图

### 前台形态

- 在 `阵容` 之外新增一个更通用的“我的炉子 / 仓库”区域
- 每个绑定号显示：
  - 角色名
  - 区服
  - 用途标签
  - 最近同步时间
  - 背包功法数量
- 下方展示：
  - 合并库存列表
  - 某张卡在哪个号上

### 后端数据建议

基于现有 `helperBindings` 扩展：

- `binding_label`
  - 用户自定义显示名
- `binding_tags`
  - 例如 `["warehouse","fulfillment"]`
- `binding_enabled`
  - 是否参与仓库同步
- `supports_inventory_sync`
  - 是否允许读取背包功法
- `supports_fulfillment`
  - 是否允许作为发货号
- `last_inventory_synced_at`
  - 最近一次背包同步时间

新增：

- `helperInventorySnapshots`
  - 某个绑定号在某次同步时的背包快照
- `helperInventoryItems`
  - 快照里的功法项明细

### 一期验收标准

- 同一个商城账号可见多个绑定角色
- 每个角色都能单独同步背包功法
- 前台能看到合并库存
- 能知道“某张卡在哪个号上”

## 阶段 2：残卷收款流水识别

目标：

- 自动记录每个炉子号收到的残卷流水
- 为订单匹配可能的收款记录
- 先做到“自动识别 + 人工确认”

### 范围

- 读取接收残卷记录
- 云端保存收款流水
- 做唯一键去重
- 与订单做候选匹配
- 后台展示“建议匹配”

### 新数据结构

- `helperTransferReceipts`
  - 接收记录原始流水
- `orderPaymentMatches`
  - 某笔订单和某条残卷记录之间的匹配关系

建议唯一键：

- `binding_id + source_role_id + amount + happened_at + remote_record_id`

### 二期验收标准

- 炉子号的收款记录能稳定落库
- 同一条收款不会重复识别
- 后台能看到“这条收款可能对应哪笔订单”

## 阶段 3：自动发货

目标：

- 收款匹配通过后，从合适的炉子号自动发功法
- 发货结果回写订单

### 范围

- 自动选择发货炉子
- 发货命令执行
- 发货结果读取
- 订单状态更新
- 失败重试 / 转人工

### 新数据结构

- `helperFulfillmentJobs`
  - 自动发货任务
- `helperFulfillmentAttempts`
  - 每次执行尝试日志

### 状态机建议

- `pending_payment_match`
- `payment_matched`
- `ready_to_fulfill`
- `fulfilling`
- `fulfilled`
- `fulfillment_failed`
- `manual_review`

### 幂等要求

- 同一条收款流水只能消费一次
- 同一笔订单只能完成一次发货
- 同一张卡不能重复分配给多个订单

## 风险点

### 1. 多设备 / 多窗口 token 一致性

现在阵容功能已经把 helper 凭证云端化。

多炉子场景下，需要继续确保：

- 每个绑定号的 token 独立保存
- 可按绑定号解析
- 不因为换设备丢失

### 2. 背包功法识别稳定性

如果功法读取接口字段不稳定，会影响库存可信度。

一期里建议：

- 先保存原始数据
- 前台展示标准化字段
- 保留调试映射能力

### 3. 收款匹配误判

自动发货前最危险的是误把别人的残卷记录认成这笔订单。

所以必须先经历：

- 自动识别
- 人工确认
- 再自动发货

### 4. 发货幂等

必须保证：

- 失败重试不会重复发
- 网络抖动不会重复扣库存

## 推荐实施顺序

1. 多炉子绑定模型
2. 背包功法同步
3. 合并库存展示
4. 残卷收款流水落库
5. 后台建议匹配
6. 半自动发货
7. 全自动发货

## 现在就开始的第一步

优先做：

- 把“单商城账号绑定一个游戏角色”扩展成“可绑定多个角色”
- 在绑定记录上加：
  - 角色用途
  - 显示名
  - 是否参与库存同步
  - 是否参与自动发货

然后再接：

- 小号绑定
- 背包功法同步
- 合并库存列表

## 是否需要小号 token

需要，而且越早越好。

原因：

- 阶段 1 做多炉子仓库时，就需要真实测试“第二个绑定号”
- 后面自动发货也一定要有可用的发货号

建议方式：

- 先把你的小号在商城里也绑上
- 如果能直接绑定到现有云端凭证最好
- 如果不行，再补一次扫码或 BIN 导入

这样我后面就能直接用这两个号开始做：

- 多绑定展示
- 多号库存同步
- 发货号选择逻辑
