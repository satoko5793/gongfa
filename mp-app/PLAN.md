# 小程序实施计划
## 目标

把当前“网页前台 + 网页后台”的结构，逐步改成：
- 小程序前台
- 网页后台
- 复用现有商城后端

当前主线仍然是网页端，小程序按独立子项目推进。

## 交付阶段

### 阶段 0：立项与目录拆分

目标：
- 把小程序工作区单独拆出来
- 不影响现有网页端开发

交付：
- `miniapp/` 独立目录
- 小程序最小骨架
- 迁移说明文档

状态：
- 已完成

### 阶段 1：商品浏览 MVP

目标：
- 小程序先具备只读浏览能力
- 不接支付，不接复杂账户逻辑

交付：
- 商品列表页
- 商品详情页
- 接现有 `/products`、`/products/:id`
- 基础的错误态、空态、加载态

涉及文件：
- `miniapp/pages/index/*`
- `miniapp/pages/product-detail/*`
- `miniapp/utils/request.js`
- `miniapp/utils/config.js`

状态：
- 已完成基础浏览链路

### 阶段 2：用户体系接入

目标：
- 小程序具备基础用户识别能力
- 可以查看自己的额度和订单
- 当前只保留最简注册字段：`游戏 ID / 游戏名称 / 密码`

交付：
- 注册/登录
- 账户页
- 我的额度
- 我的订单

决策：
- 先复用当前密码登录能力跑通 MVP
- 当前不接 helper

当前进度：
- 已接密码注册
- 已接密码登录
- 已接我的额度
- 已接我的订单

### 阶段 3：购买链路

目标：
- 在小程序里完成最小下单闭环

交付：
- 商品详情页购买确认
- 下单接口接入
- 下单后刷新额度、订单、商品库存
- 明确的下单结果提示

涉及接口：
- `POST /orders`
- `GET /me/quota`
- `GET /me/orders`

当前进度：
- 已接商品详情页下单
- 已接下单结果提示
- 已接下单后账户页自动刷新
- 已接下单后商品列表自动刷新
- 已补账户页订单状态和订单卡片展示
- 已补商品列表与详情页的基础商城样式

下一步：
- 检查小程序端异常提示和空态
- 梳理真机调试所需的接口配置

### 阶段 4：部署切换

目标：
- 让小程序前台可稳定联调

优先路线：
1. 现有后端先上微信云托管
2. 小程序直接请求云托管 API
3. 后续再评估是否迁到 CloudBase

交付：
- 统一 API 域名
- 小程序合法 request 域名配置
- 环境区分：本地 / 测试 / 线上

### 阶段 5：正式化

目标：
- 做成可真实对外测试的版本

交付：
- 更完整的视觉样式
- 登录态持久化
- 订单状态刷新策略
- 错误日志和基础埋点

## 当前优先级

当前优先推进：
1. 购买链路细节
2. 账户页体验
3. 商品样式

暂不提前做：
- 支付
- 小程序后台管理
- 微信审核适配
- 登录方案大改

## 决策记录

### 为什么不一开始就切 CloudBase

因为当前项目已经有 Node/Express 后端和成熟业务逻辑，先复用更快。

### 为什么后台继续保留网页

因为这些操作更适合网页后台：
- 导入 JSON
- 批量运营
- 商品调价
- 订单处理

### 为什么当前不接 helper

因为用户主流程已经收敛为：
- 游戏 ID
- 游戏名称
- 密码

卡牌库更新由管理员自己提取 JSON 并导入，helper 不是当前主链路必需项。
